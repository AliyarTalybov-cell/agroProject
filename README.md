## Прототип портала агронома

Этот репозиторий предназначен для фронтенд‑прототипа портала агронома.

## Запуск фронтенда

Фронтенд живёт в папке `frontend/`.

Требования: установленный Node.js (LTS).

```bash
cd frontend
npm install
npm run dev
```

## Страницы и маршруты

- **Авторизация**
  - Маршрут: `/login`
  - Назначение: вход пользователя по логину и паролю, базовая валидация, переход к дашборду после успешного входа.
- **Дашборд**
  - Маршрут: `/dashboard`
  - Назначение: сводка по хозяйству — количество полей, активные задачи, ближайшие работы, ключевые индикаторы состояния культур.
- **Поля**
  - Маршрут: `/fields`
  - Назначение: список/таблица полей с основными параметрами (название, культура, площадь, статус) и фильтрами.
- **Детали поля**
  - Маршрут: `/fields/:id`
  - Назначение: детальная информация по выбранному полю: культура, история работ, задачи, заметки, базовые показатели.
- **Задачи**
  - Маршрут: `/tasks`
  - Назначение: список задач агронома с фильтрами по статусу, дате, полю, приоритету.
- **Отчёты**
  - Маршрут: `/reports`
  - Назначение: набор отчётных экранов (урожайность по культурам, выполненные работы по периодам и т.п.).

## Выбор технологического стека

- **Основной фреймворк**: Vue 3
- **Сборщик/бандлер**: Vite
- **Язык**: TypeScript (приоритетно; при необходимости можно начать с JavaScript)
- **Состояние**: локальное состояние компонентов, при росте сложности — Pinia
- **Стили**: чистый CSS/SCSS, при необходимости можно добавить лёгкий UI‑кит (Naive UI, Element Plus и т.п.)

Таким образом, **Vue 3 + Vite** зафиксированы как базовый стек для реализации всего прототипа.

## Базовые модели данных

- **Поле (`Field`)**
  - `id`: уникальный идентификатор поля
  - `name`: название поля
  - `cropId`: ссылка на текущую культуру
  - `areaHa`: площадь в гектарах
  - `region`: регион/хозяйство
  - `status`: статус поля (например, `планируется`, `в работе`, `убрано`)
  - `lastInspectionDate`: дата последнего осмотра
  - `notes`: произвольные заметки
- **Культура (`Crop`)**
  - `id`: уникальный идентификатор культуры
  - `name`: название культуры (пшеница, кукуруза и т.п.)
  - `variety`: сорт/гибрид (опционально)
  - `season`: сезон/год (например, `2026`)
- **Задача (`Task`)**
  - `id`: уникальный идентификатор задачи
  - `fieldId`: ссылка на поле
  - `title`: краткое название задачи
  - `description`: подробное описание
  - `type`: тип задачи (обработка, осмотр, внесение удобрений и т.п.)
  - `status`: статус (`новая`, `в работе`, `завершена`, `отменена`)
  - `priority`: приоритет (`низкий`, `средний`, `высокий`)
  - `assignee`: ответственный (ФИО или роль)
  - `dueDate`: срок выполнения
  - `completedAt`: дата фактического завершения (опционально)
- **Отчёт/Сводка (`Report`)**
  - `id`: уникальный идентификатор отчёта
  - `type`: тип отчёта (урожайность по культурам, работы по периодам и т.п.)
  - `periodFrom` / `periodTo`: период, по которому строится отчёт
  - `fields`: набор полей/идентификаторов полей, попадающих в отчёт
  - `data`: агрегированные показатели (структура зависит от типа отчёта)

Связи:

- одно **поле** связано с одной актуальной **культурой** (`Field.cropId → Crop.id`);
- у одного **поля** может быть много **задач** (`Task.fieldId → Field.id`);
- **отчёты** агрегируют данные по нескольким полям, культурам и задачам.

## Чат (Supabase)

Сообщения хранятся в БД, участники — реальные пользователи из `profiles` / `auth.users`.

1. Примените `supabase/migrations/add_chat_threads_and_messages.sql` (в конце файла есть `mark_chat_thread_read`). Если чат вы добавляли раньше **без** этой функции — выполните отдельно `add_mark_chat_thread_read_rpc.sql`.
2. Если в UI ошибка **`infinite recursion detected in policy for relation "chat_thread_members"`** — выполните **`supabase/migrations/fix_chat_thread_members_rls_recursion.sql`** (или заново примените обновлённый `add_chat_threads_and_messages.sql` на чистой БД).
3. В проекте задайте `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY` для фронтенда.
4. Маршрут **`/chat`** доступен всем авторизованным пользователям; пункт **«Сотрудники»** по-прежнему только у роли `manager`.
5. **Личные диалоги**: RPC `get_or_create_dm_thread` создаёт поток `direct` между двумя пользователями.
6. **Группы («Команды»)**: RPC `create_group_thread` — только для `profiles.role = 'manager'`.
7. **Непрочитанные**: по полю `last_read_at` в `chat_thread_members`; бейдж в меню — RPC `chat_total_unread`.
8. **Realtime**: миграция добавляет `chat_messages` в публикацию `supabase_realtime` (если таблица уже в публикации, блок в миграции просто пропустит ошибку).
9. **Удаление своих сообщений**: миграция **`supabase/migrations/add_chat_messages_delete_policy.sql`** (RLS `DELETE` + `grant delete`).
10. **Вложения в чате**: если бакет **`chat-attachments`** создали вручную в Dashboard, примените только **`supabase/migrations/add_chat_attachments_storage_policies_only.sql`** (политики RLS на `storage.objects`). Иначе целиком **`add_chat_attachments_storage.sql`** (бакет + политики). Лимит файла **10 МБ** (и в бакете, и на фронте). Если бакет уже был с другим лимитом — **`update_chat_attachments_bucket_10mb.sql`**. Поля `attachment_*` в `chat_messages` уже есть в базовой миграции чата.
